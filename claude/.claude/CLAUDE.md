## LSP

- TypeScript / TSX ファイルを扱う際は、LSP ツールを優先的に使用すること
  - `hover`: 型情報の確認
  - `goToDefinition`: 定義元へのジャンプ
  - `findReferences`: 参照箇所の検索
  - `documentSymbol`: ファイル内のシンボル一覧
  - `workspaceSymbol`: ワークスペース全体のシンボル検索
- LSP の `character` パラメータはシンボル名の上に正確に配置すること（キーワードや空白上では結果が返らない）
- Grep はLSP が利用できない場合、またはテキスト・パターン検索の場合にのみ使用すること
- コードの作成・編集後は、LSP の診断結果を確認し、エラーがあれば次に進む前に修正すること

## 命名について

一般的や抽象的な名前は避け、そのドメイン領域における、担うべき目的や役割がひと目で分かる具体的な名前をつけること。

## 開発スタイル

- t-wada の推奨する TDD で開発する（探索 → Red → Green → Refactoring）。
- 指示が与えられた場合、計画のあらゆる側面について、指示者とあなたが共通の認識に達するまで、徹底的に指示者に質問を投げかけること。
  - 設計のツリーを枝分かれの先まで一つひとつたどり、決定事項間の依存関係を順番に解決していく。
  - 各質問に対し、あなたの推奨する回答も併せて提示すること。
  - もしコードベースを探索することで答えが得られる質問であれば、質問する代わりにコードベースを調査する。
- 静的検査可能なルールはプロンプトではなく、その環境の linter か ast-grep で記述する

## コード設計

- 関心の分離を保つ
- 状態とロジックを分離する
- 可読性と保守性を重視する
- コントラクト層（API/型）を厳密に定義し、実装層は再生成可能に保つ
